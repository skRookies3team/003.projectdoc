## 멘토링 내용

### 1. 프로젝트 관리 및 발표 전략 (PPT 피드백)
#### 팀 구성 및 역할 소개 (Who)
- **역할 중심 조직도**: 이름 나열 방식을 지양하고, 실제 수행한 역할을 중심으로 작성해야 함.
- **스크럼 체계 강조**: 스크럼 마스터와 각 파트별 리더(프론트, 백엔드, 배포, AI)로 구성된 조직도 그림을 통해 스크럼 체계의 R&R을 명확히 보여줄 것.

#### 방법론 및 프로세스 (How)
- **스토리라인 구성**: 주제(What) → 조직(Who) → 방법론/기술/아키텍처/설계/표준(How) → 시연 순서로 논리적 흐름 구성.
- **스크럼 운영 증명**:
  - 단순 이론 설명이 아닌 '실제 적용 사례'가 핵심.
  - 스토리 포인트 측정 방식, 문제 해결 과정, 회고를 통해 Sprint를 어떻게 발전시켰는지에 대한 구체적 데이터와 분석 제시.
  - Jira 실제 화면 캡처를 활용하여 리얼리티 부여.

### 2. 아키텍처 및 기술적 의사결정
#### 아키텍처 다이어그램 수정 (Architecture)
- **배치 순서**: 백엔드 아키텍처 → 인프라 아키텍처 → AI 아키텍처 순으로 배치.
- **백엔드 아키텍처**: 
  - 범례 필수 표기 (비동기: 점선, 동기: 실선).
  - API Gateway 등 중요도가 낮은 서비스는 뒤로 배치하거나 강조를 줄일 것.
- **인프라 아키텍처**: 
  - EKS Cluster 내 선 정리 필요 (Replica 간 연결 불필요, 경로는 합쳐서 깔끔하게).
  - Inbound/Outbound 트래픽 표시 명확화 (피드백: 매우 잘 그려짐 👍).
- **AI 아키텍처**: 
  - Front 구조 생략, Backend는 MSA로 통합하여 표현.
  - **RAG 파이프라인 강조**: Vector DB 생성 과정, 외부 LLM 호출 경로, 기록 서비스에서의 AI API 호출 흐름을 중심으로 도식화.

#### 기술 스택 및 환경 설정
- **Kafka 배포 전략**: Local 구동보다는 Docker Image를 활용하여 ArgoCD 등과 연동하는 방식 권장.
- **Redis 도입 여부**: 사용하지 않는다면 과감히 제외하거나, 사용 목적을 명확히 하여 결정 필요.
- **시각화**: 기술 스택 및 AI 관련 장표에는 아이콘을 적극 활용하여 직관성 높이기.

### 3. 발표 자료(PPT) 작성 가이드
- **가독성 및 효율성**: 
  - 글은 간결하게 줄이고, 한눈에 들어오는 그림(도식) 위주로 구성.
  - UI Framework 등은 간단히 언급하고 효율적인 장표 구성에 집중.
- **표현력**: 
  - 스프린트 일정은 타임라인(동그라미 그림 등)을 활용하여 시각화.
  - 육하원칙에 따라 관리(애자일), 설계(아키텍처), 결과가 드러나도록 설계.

## 학습 내용 및 인사이트
- **새로 배운 내용**: 
  - **발표의 본질**: 단순히 무엇을 만들었냐가 아니라, '어떤 문제'를 '어떻게(How)' 해결했는지를 보여주는 것이 핵심이다. 특히 이론(스크럼 등)을 실제 프로젝트에 어떻게 녹여냈는지를 증명하는 것이 중요하다.
  - **아키텍처의 언어**: 선의 종류(실선/점선) 하나에도 동기/비동기의 의미가 담겨야 하며, 보는 사람이 직관적으로 이해할 수 있도록 심플하게 그려야 한다.

- **핵심 인사이트**: 
  - "문서 작성과 발표 능력은 곧 연봉이다." 개발 실력만큼이나 자신을 PR하고 결과를 포장하여 전달하는 능력이 엔지니어의 가치를 높인다.
  - RAG 아키텍처에서는 단순 구조보다 데이터의 흐름(Vector DB 구축 ~ LLM 호출)이 더 중요한 설명 포인트다.
