- 헬스케어 연동: 'WITHaPET' 웨어러블 디바이스를 연동하여 실제 건강 데이터를 챗봇과 대시보드에 활용할 계획 공유.

UI 프로토타입(V0) 워크스루

- '펫 메이트' 기능 피드백: 멘토가 '실시간 GPS 기반' 매칭의 맹점(견주는 직장, 개는 집에 있는 상황)을 지적. '당근마켓'처럼 사용자가 '활동 지역'을 직접 설정하는 방식으로 변경할 것을 강력히 권고.
- '헬스케어' 기능 피드백: 'AIA 바이탈리티'의 보험 연계 사례처럼 비즈니스 모델 확장 가능성을 제안받음.
- AI 스튜디오: 'AI 다이어리'(반려동물 관점) 및 'AI 리캡'(영상 요약) 기능 논의.

요구사항 명세서 집  중 피드백
- 지속 가능한 개발: "미완료 작업을 주말에 한다"는 규칙을 멘토가 '지속 불가능한 개발'이라며 강하게 비판. 애자일 철학에 맞게, 팀의 실제 '속도(Velocity)'(예: 20점 계획, 10점 완료 시 속도는 10점)를 측정하여 다음 스프린트 계획량을 조절하는 방식을 권고.
- 스크럼(일일 회의) 방식: 비효율적인 회의록 작성 대신, 매일 아침 '기분'을 5점 척도로 공유(예: "어제 엄마와 싸워서 1점")하여 팀원 간 상태를 파악하는 방식을 제안받음.

- 회고 (KPT): 스프린트 종료 시, '잘한 점(Keep)', '개선할 점(Problem)', '시도할 점(Try)'을 포스트잇에 적어 투표하고 논의하는 회고 방식 제안.

### 2. 주요 논의 사항
#### 프로젝트 주제 선정
- **내용**: 
  - 다양한 아이디어 발산 및 토론 진행
  - 멘토님의 실무 경험 기반 피드백을 통해 주제 범위 구체화 및 축소

#### MSA(Microservice Architecture) 기초
- **주제**: 
  - MSA의 정의 및 핵심 철학
- **내용**: 
  - "작고 독립적인 서비스들의 집합"으로서의 MSA 정의
  - 모놀리식 대비 장점: 독립적 배포/관리, 유연한 변화 대응
  - 실무 적용 시 고려해야 할 현실적인 문제점들
- **학습 포인트**: 
  - MSA는 단순한 기술 트렌드가 아닌, 특정 문제를 해결하기 위한 아키텍처 패턴임

#### '문제 중심' 아키텍처 설계
- **주제**: 
  - 기술보다 문제에 집중하기
- **내용**: 
  - 기술 스택(WHAT)보다 해결하려는 문제(WHY)를 먼저 정의해야 함
  - 문제를 명확히 이해할 때 가장 효과적이고 효율적인 아키텍처 설계 가능
- **학습 포인트**: 
  - "기술 자체에 매몰되지 말자."

#### 결합도(Coupling)와 응집도(Cohesion)
- **주제**: 
  - 좋은 MSA 설계를 위한 핵심 원칙
- **내용**: 
  - 서비스 간 결합도는 낮추고(Loose Coupling), 서비스 내 응집도는 높여야 함
  - 서비스 인터페이스(API)의 명확한 정의와 책임 분리의 중요성

#### 기술 스택과 자동화
- **주제**: 
  - MSA 구현을 위한 도구 선택
- **내용**: 
  - Spring Cloud, Kubernetes, Istio 등 다양한 도구 소개
  - 모든 도구를 쓸 필요는 없으며, 프로젝트 상황에 맞게 현명하게 선택해야 함
  - 복잡성 관리를 위한 CI/CD 등 자동화의 필수성

## 학습 내용 및 인사이트
- **새로 배운 내용**: 
  - '이벤트 스토밍' 워크숍 진행 방법론 및 각 포스트잇 색깔의 명확한 의미(커맨드, 이벤트, 어그리게잇 등)를 배움.

  - 애자일 '속도(Velocity)' 개념과 '지속 가능한 개발'의 중요성을 배움.

  - MSA 도메인을 '핵심/일반/지원' 3가지로 분류하는 방법.
- **핵심 인사이트**: 
  - 요구사항 명세서는 '고객 관점(What)'으로, 아키텍처는 '기능/도메인 중심'으로 설계해야 한다. UI 프로토타입(V0)이 'What'을 정의하는 데 도움이 될 수 있다.

  - '펫 메이트'의 GPS 기반 매칭은 맹점이 있으며(견주는 직장, 개는 집), '당근마켓'의 '활동 지역' 설정 방식이 더 현실적인 모델이다.

  - '디렉토리 구조'가 단순한 폴더 정리가 아니라, 팀의 아키텍처(예: 레이어드 아키텍처) 철학을 반영하는 중요한 설계 표준임을 인식함.

  - 데이터 정책(예: '펫 삭제')을 결정할 때, 기술적 관점(Delete)이 아닌 사용자 경험(추억 보존) 관점(Update)에서 접근해야 함.


- **실무 적용 방안**: 
  - 스크럼 회의 시 '기분 척도' 공유를 도입하여 팀원 간 소통을 활성화한다.
  - '이벤트 스토밍 맵'을 기반으로 MSA별 API 설계를 시작한다.
