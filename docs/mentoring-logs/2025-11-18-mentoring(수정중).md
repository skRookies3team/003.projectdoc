- MSA 서비스 경계(Bounded Context) 설정에 대한 명확한 기준 부재
  
### 2. 주요 논의 사항

#### 기술적 내용
- **주제**: 
  - '문제 중심' 요구사항 정의 및 MSA 설계 방법론 (이벤트 스토밍)
- **내용**: 
  - 요구사항 정의: 'How(어떻게 구현할지/페이지)'가 아닌 'What(무엇을 원하는지/비즈니스 가치)' 관점에서 작성해야 함.
  - MSA 설계: 기술 스택(Tech Stack)보다 비즈니스 도메인(Domain) 경계를 먼저 정의해야 함. 이를 위해 '이벤트 스토밍' 워크숍을 진행.
  - 이벤트 스토밍:
    - 커맨드(Command): 이벤트를 유발하는 트리거 (API 요청 등)
    - 이벤트(Event): 비즈니스 상태의 변화 (과거형 기술)
    - 어그리게잇(Aggregate): 데이터의 묶음 (DB 엔티티 후보)
    - 바운디드 컨텍스트(Bounded Context): 연관된 기능을 그룹화하여 마이크로서비스 후보 도출
- **학습 포인트**: 
  - 요구사항은 고객의 언어로 '기능' 중심으로 정의해야 하며, UI 설계와는 분리해야 한다.
  - MSA 아키텍처는 기술적 판단 이전에 비즈니스 프로세스 분석을 통해 도출된다.

#### 프로젝트 관련
- **현재 진행 상황**: 
  - 기획: '펫 페르소나 챗봇', 'WITHaPET 헬스케어 연동', '펫 쇼핑몰' 등 핵심 기능 확정.
  - 설계: 이벤트 스토밍을 통해 6개 도메인(회원, 펫, 소셜, 기록, 쇼핑, 헬스케어)으로 서비스 분리 완료.
- **다음 단계 계획**: 
  - 도출된 6개 도메인별로 Git Repository 생성 및 디렉토리 구조 표준화.
  - 이벤트 스토밍 결과(커맨드/어그리게잇)를 바탕으로 상세 API 명세서 작성.
  - 재작성된 '도메인 중심' 요구사항 명세서 완료.
- **예상 어려움**: 
  - 펫 메이트: 실시간 GPS 매칭의 현실적 한계(견주 위치 vs 펫 위치) -> '활동 지역' 설정 방식으로 변경 필요.
  - 데이터 정책: 펫 정보 삭제 시 연관 데이터(다이어리 등) 처리 문제 -> '상태 변경(Update)'으로 보존 정책 결정.
  - AI 연동: RAG 구현 등 AI 기술의 구체적인 적용 방안 수립 필요.

### 3. 애자일 및 협업 방식 (그라운드 룰)
- **스크럼 회의**: 
  - 단순 업무 보고가 아닌, 매일 아침 팀원의 '기분(5점 척도)'을 공유하여 상호 상태를 파악하고 소통을 강화하기로 함.
- **지속 가능한 개발**: 
  - "주말 근무"와 같은 무리한 계획 대신, 팀의 실제 '속도(Velocity)'를 측정하여 현실적인 스프린트 계획을 수립하기로 함.
- **회고**:
  - 스프린트 종료 시 KPT(Keep, Problem, Try) 방식의 회고 도입.

## 학습 내용 및 인사이트
- **새로 배운 내용**: 
  - '이벤트 스토밍' 워크숍 진행 방법론 및 각 포스트잇 색깔의 명확한 의미(커맨드, 이벤트, 어그리게잇 등)를 배움.

  - 애자일 '속도(Velocity)' 개념과 '지속 가능한 개발'의 중요성을 배움.

  - MSA 도메인을 '핵심/일반/지원' 3가지로 분류하는 방법.
- **핵심 인사이트**: 
  - 요구사항 명세서는 '고객 관점(What)'으로, 아키텍처는 '기능/도메인 중심'으로 설계해야 한다. UI 프로토타입(V0)이 'What'을 정의하는 데 도움이 될 수 있다.

  - '펫 메이트'의 GPS 기반 매칭은 맹점이 있으며(견주는 직장, 개는 집), '당근마켓'의 '활동 지역' 설정 방식이 더 현실적인 모델이다.

  - '디렉토리 구조'가 단순한 폴더 정리가 아니라, 팀의 아키텍처(예: 레이어드 아키텍처) 철학을 반영하는 중요한 설계 표준임을 인식함.

  - 데이터 정책(예: '펫 삭제')을 결정할 때, 기술적 관점(Delete)이 아닌 사용자 경험(추억 보존) 관점(Update)에서 접근해야 함.


- **실무 적용 방안**: 
  - 스크럼 회의 시 '기분 척도' 공유를 도입하여 팀원 간 소통을 활성화한다.
  - '이벤트 스토밍 맵'을 기반으로 MSA별 API 설계를 시작한다.
