### 안건 2: JWT 토큰 관리 및 저장소 전략 수립

**논의 내용:**
- **토큰 저장 방식 논의:**
  - `localStorage`: XSS 공격 취약점 존재하나 구현이 용이하고 새로고침 시 데이터 유지가 가능.
  - `Cookie (HttpOnly)`: 보안상 우수하나, 구현 복잡도 증가 및 서버 프록시 설정 필요 (BFF 패턴 등).
  - **Context API:** 전역 상태 관리에 유용하나, 새로고침 시 데이터(로그인 상태)가 초기화되는 문제 발생.
- **UX(사용자 경험) 고려:** 새로고침 시 로그아웃되는 현상은 사용자에게 큰 불편을 주므로 지속성(Persistence) 유지가 필수적임.
- **보안 보완책:** Access Token의 만료 시간을 짧게 설정하고, Refresh Token을 통한 자동 갱신 로직을 도입하여 보안 위협을 완화.

**결정 사항:**
1. **저장소:** 사용자 편의성과 유지보수 용이성을 고려하여 **LocalStorage** 사용 결정.
2. **상태 관리:** `Context API`를 사용하여 전역에서 토큰 접근이 가능하도록 구현하되, 새로고침 시 LocalStorage에서 값을 가져와 상태를 복구하는 로직 추가.
3. **보안 강화:** Access Token 만료 시 Refresh Token을 사용해 토큰을 자동 갱신하는 로직(Silent Refresh)을 구현하여 보안성 확보.
4. **구현 상세:**
   - Axios Interceptor(`config.headers.Authorization`)를 활용하여 API 요청 시 자동으로 토큰 주입.
   - 강사님 제공 코드(`getTokenFunction`) 및 Axios 인스턴스 패턴 참고.

**담당자:** 신재석, 김현근 (FE/BE 협업)
